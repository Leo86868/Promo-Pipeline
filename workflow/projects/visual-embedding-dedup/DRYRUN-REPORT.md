**日期** 2026-06-25 · **分支** `feat/visual-diversity-select`（未 push）

## 一句话
今天下载的 30 条里，有 **5–15 对视觉近重复**、只有 **19–25 个真正不同的画面簇**（30 个名额里 5–11 个浪费在重复上）。把"挑哪 30"换成 **相关性种子 + 视觉 max-min**，**窗口=45 时每个 POI 都干到 0 近重复 / 30 个不同簇**，下载数仍是 30（egress 不变）。

## 方法（消费侧，对齐 handoff + prototype）
- **基线 relevance-30** = 按相关性取前 30（今天的选法；今天还会用"低用量储备"补位，这里取纯相关性版作对比）。
- **maxmin@W** = 相关性 top-W 当考虑池 → 贪心最远点选 30，**种子=相关性#1**（不是 prototype 的 medoid，PGC 适配：保证最相关那条必在）。
- **白化** = 对 top-60 工作池做 mean-center → L2（handoff"消费侧白化"）；所有选法在同一白化空间里比，公平。
- **指标**：近重复对(≥0.85 / ≥0.92 白化余弦) · worst-pair(集合内最大余弦) · 不同簇数(单链 @0.85) · deepest_rk(摸到第几名相关性) · top15_kept(前15相关里留几个=召回代理)。

⚠️ **相关性是代理**：OpenRouter key **401**（环境里那把已轮换失效）→ 没法离线重嵌真脚本 query；本地也没有 DB 凭据，只有只读 MCP。所以相关性用 **文字向量到 POI 文字质心的余弦**（最有代表性的排前）代替。代理可能**略微高估基线的扎堆**（质心偏向模态场景）；但 **maxmin 一侧的结果（@45 全 0 近重复）跟相关性代理无关**，是选法本身的性质，稳。基线扎堆是真的——①已独立证过真视觉双胞胎存在（OKANA 0.974 等）。

## 读数
1. **今天确有视觉浪费**：基线 5–15 对近重复、1–6 对近乎同一镜头(≥0.92)、worst 0.93–0.97（=两条几乎一样的片同时进下载池）、只 19–25 个不同簇。Clear Sky Domes 最惨（穹顶都长一样：15 对、19 簇）。
2. **@35 窗口太窄**：有改善但没清干净（还剩 0–5 对）——池子没腾开。
3. **@45 是拐点**：5 个大池 POI **全部** → 0 近重复 + 30/30 不同簇，worst 掉到 0.63–0.79。召回代价中等（前15相关留 8–10，摸到 ~rank 45）。
4. **@60 边际收益小、代价大**：worst 再降一点点，但前15相关只留 5–9、摸到 rank 57–60 → 相关性割肉换不到多少多样性。
5. **小池优雅退化**：池=34 时 @45 仍改善(5→2 对)但清不干净（没空间）；**从不变差**（仍下 30）。→ flag 在大池强、小池温和、处处安全。

## 建议
**窗口 = 45**（=拐点：全清近重复 + 满簇覆盖，召回代价可接受；比 60 省相关性、比 35 干净）。
近重复阈值沿用 **0.85**（跟①一致；0.92 留给"只删同一镜头"的更保守口味）。
